# Project Description — CUDA GPU Image Processing Pipeline

## Purpose

This project demonstrates GPU-accelerated batch image processing by implementing
a 5-stage pipeline that transforms colour PNG images into several processed
variants entirely on the GPU. The goal is to show how CUDA — both through the
high-level NPP library and hand-written kernels — can process large volumes of
image data far faster than a CPU-only implementation.

## Algorithms and Kernels

### Stage 1 — RGB to Grayscale (NPP)
Uses `nppiRGBToGray_8u_C3C1R` from the CUDA NPP library. NPP applies the
standard luminance formula (0.299R + 0.587G + 0.114B) across all pixels in
parallel on the GPU. No host-side loops are involved after the initial
host→device memcpy.

### Stage 2 — Gaussian Blur (NPP)
Uses `nppiFilterGaussBorder_8u_C1R` with a 5×5 mask and border replication.
Gaussian blurring reduces high-frequency noise and is a prerequisite for
robust edge detection. NPP's implementation uses separable convolution
internally, making it highly efficient on the GPU.

### Stage 3 — Sobel Edge Detection (NPP)
Uses `nppiFilterSobelHorizBorder_8u_C1R` on the blurred grayscale image.
The Sobel operator approximates the horizontal image gradient, highlighting
edges and transitions. Applying it after Gaussian blur (the classic
"Gaussian → Sobel" pipeline) reduces noise-induced false edges.

### Stage 4 — Brightness and Contrast Adjustment (Custom Kernel)
Custom kernel `BrightnessContrastKernel` applies the linear transformation:

    dst[i] = clamp(alpha * src[i] + beta, 0, 255)

where `alpha` controls contrast and `beta` controls brightness. Each thread
handles one pixel, so the kernel launches a 2D grid covering the entire image.
This demonstrates writing CUDA device code directly rather than relying on a
library.

### Stage 5 — Image Inversion (Custom Kernel)
Custom kernel `InvertKernel` applies:

    dst[i] = 255 - src[i]

Simple but effective for demonstrating that multiple independent kernel
operations can be chained on the same device buffer without returning to the
host between stages.

## Dataset

Test images are generated by `scripts/generate_test_images.py`, a pure-Python
script (no third-party libraries) that creates 20 synthetic 512×512 PNG images
covering ten different patterns:

- Linear gradients (two angles)
- Concentric circles
- Checkerboard (ideal for edge detection)
- Random noise
- Horizontal and vertical stripes
- Sine-wave pattern
- Radial gradient
- Diagonal lines

These patterns make the visual effect of each pipeline stage immediately
obvious, and the variety ensures that all code paths (e.g. bright regions,
dark regions, high-frequency content) are exercised.

## GPU vs CPU

The NPP library operations (grayscale conversion, Gaussian blur, Sobel) are
executed entirely on the GPU — data travels host→device once at the start and
device→host once at the end per image. No intermediate results are copied back
to the CPU between stages. The two custom kernels also execute in-place on the
GPU. This design minimises PCIe transfer overhead and maximises GPU utilisation.

## Lessons Learned

1. **NPP border handling is critical.** The first working version used
   `nppiFilterGauss` (without border handling), which requires the ROI to be
   smaller than the image by the kernel radius. Switching to
   `nppiFilterGaussBorder` with `NPP_BORDER_REPLICATE` simplifies the ROI
   management and avoids off-by-one errors at image edges.

2. **libpng is sufficient and universally available.** The original design used
   the stb_image header-only library, but downloading single-file headers from
   GitHub during a restricted lab environment proved unreliable. libpng is
   installed on every CUDA-equipped Linux machine and links cleanly with nvcc.

3. **Pure-Python PNG generation removes all data dependencies.** Writing a
   minimal PNG encoder using only `zlib` and `struct` from the Python standard
   library means test image generation works in any environment without
   Pillow, OpenCV, or any other package — important for reproducibility.

4. **Synthetic data beats real data for unit testing.** Checkerboard and
   diagonal-line images produce highly predictable Sobel outputs, making it
   easy to visually verify correctness at a glance. Random noise images test
   the pipeline under worst-case memory-access patterns.

5. **CUDA arch flags matter.** Compiling with the wrong `-arch` flag produces
   code that either refuses to run or silently falls back to a slower PTX
   JIT path. Documenting the correct flag for each GPU generation in the
   README saves significant debugging time.

## Significance

Processing 20 images through a 5-stage GPU pipeline (100 output files) in
under one second on a single consumer GPU illustrates the throughput advantage
of GPU computing for embarrassingly parallel workloads like image processing.
Scaling to hundreds or thousands of images requires only changing the contents
of the input directory — the pipeline is fully data-driven with no hard-coded
image counts.
